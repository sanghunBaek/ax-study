# Research: 내 기록 당첨 여부 표기

## 1. 타임스탬프 → 회차 매핑 전략

**결정**: DB에서 `drw_date` 기준으로 매핑한다.

**근거**:
- 기록의 `ts`(밀리초 타임스탬프)를 KST 날짜로 변환
- 해당 날짜 이후 가장 가까운 토요일의 회차를 찾는다
- 토요일 21:00 이후 저장분은 다음 주 토요일 회차로 매핑
- Supabase에서 `drw_date >= 기준일` 조건으로 1건 조회하면 됨

**대안 검토**:
- 고정 공식 계산(1회차 기준일 + 주차 계산): 공휴일/미추첨 주 존재 시 오차 발생 가능 → 기각
- 프론트에서 전체 회차 목록 캐시 후 매핑: 데이터 양이 많아 비효율적 → 기각

**최종 접근**: 기록별 `ts`에서 "다음 토요일" 날짜를 계산하고, DB에서 해당 `drw_date`의 회차를 조회. DB에 없으면 미발표로 처리.

## 2. 당첨 등수 판정 로직

**결정**: 프론트엔드에서 순수 함수로 판정한다.

**근거**:
- 로또 등수 규칙은 고정적이고 단순함 (6개 번호 비교)
- DB 조회는 당첨번호만 가져오면 됨, 별도 RPC 불필요
- 판정 로직: matchCount(일치 개수) + bonusMatch(보너스 일치 여부)로 등수 결정

**등수 판정 규칙**:

|---------|------------|------|
| 6       | -          | 1등  |
| 5       | O          | 2등  |
| 5       | X          | 3등  |
| 4       | -          | 4등  |
| 3       | -          | 5등  |
| 0~2     | -          | 낙첨 |

## 3. Supabase 조회 최적화

**결정**: 기록에 필요한 회차만 배치 조회한다.

**근거**:
- 기록 최대 200건이지만, 고유 회차 수는 훨씬 적음 (보통 수십 개)
- 고유 회차 번호를 추출 후 `drw_no IN (...)` 조건으로 한 번에 조회
- 회차 매핑이 안 되는 기록(미래 날짜)은 조회 대상에서 제외

**대안 검토**:
- 기록별 개별 조회: N+1 문제 → 기각
- 전체 회차 조회: 1200건+ 불필요한 데이터 → 기각

## 4. 회차 매핑을 위한 DB 조회 방식

**결정**: 프론트에서 "다음 토요일" 날짜를 계산한 뒤, `drw_date`로 회차를 조회한다.

**접근**:
1. `ts` → KST Date 변환
2. 토요일 21:00 이전이면 해당 주 토요일, 이후면 다음 주 토요일
3. 계산된 토요일 날짜로 `lottery_draws` 테이블에서 `drw_date` 매칭
4. 매칭되는 회차가 없으면 → 미발표

**이유**: `drw_date`는 실제 추첨일이므로 공휴일 등으로 추첨이 밀린 경우에도 정확함. 다만, 기록의 토요일이 실제 추첨일과 다를 수 있으므로(추첨 미실시 주), 이 경우 미발표로 처리하는 것이 안전함.

## 5. 뱃지 색상 체계

**결정**: 등수별 구분 색상을 사용한다.

| 상태 | 배경색 | 텍스트색 | Hex 코드 |
|------|--------|---------|----------|
| 1등 | 금색 | 진한 갈색 | bg: #FFD700, fg: #7A5C00 |
| 2등 | 은색 | 진한 회색 | bg: #C0C0C0, fg: #4A4A4A |
| 3등 | 동색 | 흰색 | bg: #CD7F32, fg: #FFFFFF |
| 4등 | 파란색 | 흰색 | bg: #0066FF, fg: #FFFFFF |
| 5등 | 연한 파란색 | 파란색 | bg: #EAF2FE, fg: #0066FF |
| 낙첨 | 연한 회색 | 회색 | bg: #F2F3F5, fg: #A8A8AB |
| 미발표 | 연한 회색 | 회색 | bg: #F2F3F5, fg: #C2C4C8 |

**근거**: 기존 앱의 색상 팔레트(BLUE=#0066FF, INK=#171719, SUB=#6A6B6F)와 조화. 1~3등은 전통적 금/은/동으로 직관적 인식. 4-5등은 앱의 메인 컬러인 파란색 계열.
