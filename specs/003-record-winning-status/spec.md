# Feature Specification: 내 기록 당첨 여부 표기

**Feature Branch**: `003-record-winning-status`
**Created**: 2026-05-13
**Status**: Draft
**Input**: User description: "내 기록 페이지에서 등록한 번호의 당첨 여부를 표기. 미발표/1등~5등/낙첨 상태를 우측에 표시"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 발표된 회차의 당첨 여부 확인 (Priority: P1)

사용자가 내 기록 탭에 진입하면, 각 기록 카드의 우측에 당첨 결과가 표시된다. 이미 추첨이 완료된 회차에 대해서는 1등~5등 또는 낙첨 여부가 뱃지 형태로 보인다.

**Why this priority**: 핵심 가치 — 사용자가 자신이 뽑은 번호의 당첨 여부를 한눈에 확인할 수 있어야 한다.

**Independent Test**: 과거 회차 데이터가 있는 상태에서 기록을 저장한 뒤, 내 기록 탭에서 당첨 결과 뱃지가 올바르게 표시되는지 확인

**Acceptance Scenarios**:

1. **Given** 사용자가 번호 6개를 저장한 기록이 있고 해당 주 추첨이 완료됨, **When** 내 기록 탭에 진입, **Then** 해당 기록 카드 우측에 등수(1등~5등) 또는 "낙첨" 뱃지가 표시된다
2. **Given** 사용자의 번호 중 6개 모두 당첨번호와 일치, **When** 내 기록 탭에 진입, **Then** "1등" 뱃지가 표시된다
3. **Given** 사용자의 번호 중 5개가 당첨번호와 일치하고 나머지 1개가 보너스 번호와 일치, **When** 내 기록 탭에 진입, **Then** "2등" 뱃지가 표시된다
4. **Given** 사용자의 번호 중 일치하는 번호가 2개 이하, **When** 내 기록 탭에 진입, **Then** "낙첨" 뱃지가 표시된다

---

### User Story 2 - 미발표 회차 상태 표시 (Priority: P1)

사용자가 아직 추첨이 진행되지 않은 회차의 기록을 볼 때, "미발표" 상태가 표시된다.

**Why this priority**: 미발표 상태를 명확히 보여줘야 사용자가 혼동하지 않는다. P1과 동일하게 핵심 기능.

**Independent Test**: 최신 회차 이후에 저장된 기록이 "미발표"로 표시되는지 확인

**Acceptance Scenarios**:

1. **Given** 사용자가 번호를 저장했고 아직 해당 주 추첨이 진행되지 않음, **When** 내 기록 탭에 진입, **Then** 해당 기록 카드 우측에 "미발표" 뱃지가 표시된다
2. **Given** 추첨이 완료되어 DB에 결과가 반영됨, **When** 내 기록 탭을 다시 방문, **Then** "미발표"가 실제 등수 또는 "낙첨"으로 변경된다

---

### Edge Cases

- 사용자가 같은 날 여러 번 번호를 뽑은 경우, 각 기록마다 독립적으로 당첨 여부가 표시되어야 한다
- DB에서 당첨 데이터를 가져오지 못할 경우(네트워크 오류 등), 당첨 여부 대신 "확인 불가" 또는 별도 오류 표시 없이 "미발표"로 표시한다
- 기록 저장 시점과 추첨일 매핑: 일요일~토요일 21:00(KST) 사이에 저장된 기록은 해당 주 토요일 회차에 매핑한다. 토요일 21:00 이후~다음 일요일 사이에 저장된 기록은 다음 주 토요일 회차에 매핑한다

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 각 기록의 저장 시점을 기준으로 해당하는 추첨 회차를 자동 결정해야 한다 (해당 주 토요일 회차)
- **FR-002**: 시스템은 DB에서 해당 회차의 당첨 번호(num1~num6, bonus)를 조회하여 사용자 번호와 대조해야 한다
- **FR-003**: 시스템은 로또 당첨 등수를 정확히 판정해야 한다 (1등: 6개 일치, 2등: 5개+보너스, 3등: 5개, 4등: 4개, 5등: 3개, 그 외: 낙첨)
- **FR-004**: 아직 추첨되지 않은 회차의 기록은 "미발표" 상태로 표시해야 한다
- **FR-005**: 당첨 결과는 기록 카드의 우측에 뱃지 형태로 표시해야 한다
- **FR-006**: 당첨 결과 뱃지는 등수별로 구분된 색상을 사용해야 한다 (1등=금색, 2등=은색, 3등=동색, 4-5등=파란색, 낙첨=회색, 미발표=연한 회색)
- **FR-007**: 당첨 여부 판정에 필요한 데이터는 Supabase에서 조회하며, 조회 실패 시 해당 기록은 "미발표"로 표시한다
- **FR-008**: 기록 카드에 해당 추첨 회차 번호를 날짜 옆에 표시해야 한다 (예: "1223회 · 5/10 · 14:30")

### Key Entities

- **LottoRecord (기존)**: 사용자가 뽑은 번호 기록 (id, ts, mode, nums) — localStorage 저장
- **lottery_draws (기존)**: 추첨 결과 데이터 (drw_no, drw_date, num1~num6, bonus) — Supabase DB
- **WinningResult (신규, 런타임)**: 기록별 당첨 판정 결과 (rank: 1~5 | 'miss' | 'pending', matchCount, bonusMatch)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자가 내 기록 탭에서 1초 이내에 모든 기록의 당첨 여부를 확인할 수 있다
- **SC-002**: 당첨 등수 판정이 로또 공식 규칙과 100% 일치한다
- **SC-003**: 미발표/당첨/낙첨 상태가 시각적으로 명확히 구분되어 사용자가 즉시 인식할 수 있다
- **SC-004**: 기존 기록 화면의 레이아웃과 일관된 디자인을 유지한다

## Clarifications

### Session 2026-05-13

- Q: 당첨 뱃지 색상 구분 방식 → A: 등수별 다른 색상 (1등=금, 2등=은, 3등=동, 4-5등=파랑, 낙첨=회색, 미발표=연한 회색)
- Q: 회차 번호 표시 여부 → A: 회차 번호를 날짜 옆에 함께 표시 (예: "1223회")

## Assumptions

- 기록의 저장 시점(ts)을 기준으로 해당 주 토요일 추첨 회차를 매핑한다. 토요일 21:00(KST) 이후 저장된 기록은 다음 주 회차로 간주한다.
- Supabase DB에 최신 회차까지의 당첨 데이터가 적재되어 있다고 가정한다 (`/check-api`로 동기화 관리).
- 당첨 결과 조회는 내 기록 탭 진입 시 한 번 수행하며, 실시간 업데이트는 하지 않는다 (탭 재진입 시 갱신).
- 기존 localStorage에 저장된 과거 기록도 소급 적용하여 당첨 여부를 표시한다.
- 당첨금 정보는 표시하지 않으며, 등수만 표시한다.
