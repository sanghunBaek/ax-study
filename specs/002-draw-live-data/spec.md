# Feature Specification: 뽑기 탭 실시간 데이터 연동

**Feature Branch**: `002-draw-live-data`
**Created**: 2026-05-09
**Status**: Draft
**Input**: User description: "첫 화면 뽑기 탭에서 최근 당첨 보여주는 항목과 HOT/COLD/MIX 모드로 번호를 랜덤하게 뽑아서 번호 선택 화면에 반영하는 작업"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 최근 당첨 번호 실시간 표시 (Priority: P1)

사용자가 뽑기 탭(홈 화면)에 진입하면, 가장 최근 회차의 당첨 번호가 Supabase DB에서 자동으로 불러와져 화면 상단에 표시된다. 현재 하드코딩된 데이터 대신 실제 최신 당첨 결과를 보여준다.

**Why this priority**: 사용자가 앱을 열었을 때 가장 먼저 보는 정보이며, 신뢰감을 주는 핵심 요소. 하드코딩 데이터는 시간이 지날수록 잘못된 정보를 보여주게 됨.

**Independent Test**: 뽑기 탭 진입 시 Supabase에서 최신 회차 데이터를 조회하고, 회차 번호/당첨 번호/보너스 번호가 DB 데이터와 일치하는지 확인하여 테스트 가능.

**Acceptance Scenarios**:

1. **Given** 사용자가 앱을 실행하고 뽑기 탭에 있을 때, **When** 화면이 로드되면, **Then** Supabase DB의 최신 회차 번호, 당첨 번호 6개, 보너스 번호가 상단 카드에 표시된다.
2. **Given** 네트워크 요청 중일 때, **When** 데이터 로딩이 진행 중이면, **Then** 로딩 상태가 표시되고, 완료 후 데이터로 교체된다.
3. **Given** 네트워크 오류가 발생했을 때, **When** Supabase 조회에 실패하면, **Then** localStorage에 캐시된 마지막 성공 데이터가 표시된다. 캐시도 없으면 에러 메시지와 재시도 버튼이 표시된다.

---

### User Story 2 - HOT/COLD/MIX 번호풀 실시간 생성 (Priority: P1)

사용자가 뽑기 모드(HOT/COLD/MIX)를 선택하면, Supabase RPC 함수를 호출하여 실제 최근 회차 통계 기반의 번호풀 20개를 생성한다. 현재 가짜 빈도 데이터로 만들어진 번호풀 대신 실제 통계를 사용한다.

**Why this priority**: 앱의 핵심 가치인 "통계 기반 번호 추천"이 실제 데이터로 동작해야 의미가 있음. 하드코딩된 가짜 빈도 데이터는 앱의 존재 이유를 무의미하게 만듦.

**Independent Test**: HOT 모드 선택 시 `get_hot_numbers` RPC 호출 결과 20개 번호가 스크래치 화면에 나타나는지 확인하여 독립적으로 테스트 가능.

**Acceptance Scenarios**:

1. **Given** 사용자가 뽑기 탭에서 HOT 모드를 선택했을 때, **When** 스크래치 화면으로 진입하면, **Then** Supabase `get_hot_numbers` RPC 결과인 상위 20개 번호가 스크래치 카드에 표시된다.
2. **Given** 사용자가 COLD 모드를 선택했을 때, **When** 스크래치 화면으로 진입하면, **Then** Supabase `get_cold_numbers` RPC 결과인 하위 20개 번호가 스크래치 카드에 표시된다.
3. **Given** 사용자가 MIX 모드를 선택했을 때, **When** 스크래치 화면으로 진입하면, **Then** HOT 상위 10개 + COLD 하위 10개를 조합한 20개 번호가 스크래치 카드에 표시된다.

---

### User Story 3 - 번호풀 로딩 중 사용자 경험 (Priority: P2)

모드 선택 후 번호풀을 서버에서 불러오는 동안, 사용자에게 로딩 상태를 명확히 보여주고 데이터가 준비되면 자연스럽게 스크래치 화면으로 전환한다.

**Why this priority**: P1 기능이 동작하면 네트워크 지연이 발생할 수 있으므로, 사용자가 혼란 없이 기다릴 수 있는 피드백이 필요.

**Independent Test**: 네트워크 속도를 제한한 상태에서 모드 선택 시 로딩 인디케이터가 표시되고, 데이터 도착 후 스크래치 화면이 정상 렌더링되는지 확인.

**Acceptance Scenarios**:

1. **Given** 사용자가 모드를 선택했을 때, **When** RPC 호출이 진행 중이면, **Then** 로딩 상태를 시각적으로 표시한다.
2. **Given** RPC 호출이 완료되었을 때, **When** 번호풀 20개가 준비되면, **Then** 스크래치 화면이 해당 번호풀로 렌더링된다.
3. **Given** RPC 호출이 실패했을 때, **When** 네트워크 오류가 발생하면, **Then** 사용자에게 오류 안내와 재시도 옵션을 제공한다.

---

### Edge Cases

- Supabase RPC 호출이 타임아웃될 경우: Supabase 클라이언트 기본 타임아웃에 위임하며, 일반 네트워크 오류와 동일하게 캐시 폴백 처리
- MIX 모드에서 HOT과 COLD 번호가 겹칠 경우: 중복 번호를 제거하고 다음 순위 번호로 대체하여 20개를 채움

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 뽑기 탭 진입 시 Supabase DB에서 가장 최근 회차의 당첨 정보(회차 번호, 당첨 번호 6개, 보너스 번호)를 조회하여 표시해야 한다.
- **FR-002**: 시스템은 HOT 모드 선택 시 Supabase `get_hot_numbers` RPC를 호출하여 최근 100회차 기준 상위 출현 번호 20개를 번호풀로 사용해야 한다.
- **FR-003**: 시스템은 COLD 모드 선택 시 Supabase `get_cold_numbers` RPC를 호출하여 최근 100회차 기준 하위 출현 번호 20개를 번호풀로 사용해야 한다.
- **FR-004**: 시스템은 MIX 모드 선택 시 HOT 상위 10개 + COLD 하위 10개를 조합하여 20개 번호풀을 생성해야 한다. 중복 번호가 있을 경우 다음 순위 번호로 대체한다.
- **FR-005**: 시스템은 데이터 로딩 중 사용자에게 로딩 상태를 시각적으로 표시해야 한다.
- **FR-006**: 시스템은 네트워크 오류 시 localStorage에 캐시된 마지막 성공 데이터로 폴백해야 한다. 캐시가 없을 경우 에러 안내와 재시도 버튼을 제공해야 한다.
- **FR-007**: 시스템은 최근 당첨 정보 및 번호풀 데이터 조회 성공 시 localStorage에 캐시하여, 이후 네트워크 오류 시 폴백 데이터로 사용해야 한다.
- **FR-008**: 번호풀이 정상적으로 로드되면, 기존 스크래치 카드 화면에 해당 번호풀이 그대로 전달되어 긁기/선택 플로우가 동작해야 한다.

### Key Entities

- **최근 당첨 정보**: 회차 번호, 추첨일, 당첨 번호 6개(오름차순), 보너스 번호. Supabase `lottery_draws` 테이블에서 가장 최근 1건 조회.
- **번호풀(Number Pool)**: 모드별로 생성되는 20개 번호 배열. HOT/COLD는 Supabase RPC, MIX는 두 RPC 결과 조합으로 생성.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 뽑기 탭 진입 시 최근 당첨 번호가 3초 이내에 화면에 표시된다.
- **SC-002**: 모드 선택 후 스크래치 화면 진입까지 3초 이내에 완료된다.
- **SC-003**: 모든 모드(HOT/COLD/MIX)에서 번호풀이 정확히 20개로 생성되어 스크래치 카드에 표시된다.
- **SC-004**: 네트워크 오류 발생 시 사용자가 재시도할 수 있는 수단이 제공된다.
- **SC-005**: 기존 스크래치 → 번호 선택 → 완료 플로우가 실시간 데이터로도 정상 동작한다.

## Clarifications

### Session 2026-05-13

- Q: 네트워크 오류 시 폴백 동작 방식은? → A: localStorage에 마지막 성공 데이터를 캐시하고, 오류 시 캐시 데이터를 표시. 캐시도 없으면 에러 메시지 + 재시도 버튼.
- Q: 번호풀 20개 미만 반환 시 처리는? → A: 불가능한 상황 (1,200회차 이상 적재, 100회차 범위 내 모든 번호 출현 보장). Edge Case에서 제거.

## Assumptions

- Supabase에 `get_hot_numbers`, `get_cold_numbers` RPC 함수가 이미 등록되어 있다 (통계 화면 작업에서 설정 완료).
- `lottery_draws` 테이블에 충분한 회차 데이터(100회차 이상)가 적재되어 있다.
- 기존 스크래치 카드 UI 컴포넌트 및 번호 선택 플로우는 변경 없이 재사용한다 (번호풀만 동적으로 주입).
- `range_count` 기본값은 100으로 설정하며, 사용자 설정 변경은 이 스펙 범위 밖이다.
- 최근 당첨 정보 조회와 번호풀 RPC 호출은 독립적으로 수행되며, 최근 당첨 정보는 홈 화면 로드 시, 번호풀은 모드 선택 시 호출한다.
