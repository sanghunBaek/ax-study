# Feature Specification: 통계 페이지 실시간 데이터 연동

**Feature Branch**: `001-stats-live-data`
**Created**: 2026-05-06
**Status**: Draft
**Input**: User description: "통계 페이지(StatsScreen)의 하드코딩 데이터를 Supabase DB 실시간 데이터로 교체. 범위 필터(최근 50회, 최근 100회, 전체), HOT/COLD 번호, 1~45 빈도 바 차트, 역대 회차 목록 모두 포함."

## Clarifications

### Session 2026-05-06

- Q: 빈도 계산을 어디서 수행할 것인가? (DB vs 프론트) → A: DB(Supabase RPC)에서 빈도를 집계하여 결과만 프론트에 반환한다.
- Q: 범위 탭 전환 시 캐싱 전략은? → A: 페이지 진입 시 3개 범위(50/100/전체) 데이터를 모두 조회 후 캐싱한다. 탭 전환은 캐시된 데이터를 즉시 사용하며, 페이지 재진입 시 새로 조회한다. 별도 TTL 캐시는 사용하지 않는다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 범위별 번호 출현 빈도 조회 (Priority: P1)

사용자가 통계 페이지에 진입하면 실제 로또 추첨 데이터를 기반으로 1~45번 각 번호의 출현 빈도를 확인할 수 있다. "최근 50회", "최근 100회", "전체" 탭을 전환하면 해당 범위의 실제 빈도 데이터가 표시된다.

**Why this priority**: 통계 페이지의 핵심 기능이며, 현재 하드코딩된 가짜 데이터를 실제 데이터로 교체하는 가장 중요한 작업이다. 이것 하나만 완성해도 통계 페이지가 실질적인 가치를 제공한다.

**Independent Test**: 통계 페이지에 접속하여 범위 탭을 전환하고, 표시되는 숫자가 실제 DB의 추첨 데이터와 일치하는지 비교하여 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 사용자가 통계 페이지에 진입한 상태, **When** "최근 100회" 탭이 선택되어 있으면, **Then** DB의 최근 100회차 추첨 데이터를 기반으로 1~45번 각 번호의 출현 횟수가 바 차트에 표시된다.
2. **Given** 사용자가 "최근 100회" 탭을 보고 있는 상태, **When** "최근 50회" 탭을 클릭하면, **Then** DB의 최근 50회차 데이터 기반 빈도로 바 차트가 갱신된다.
3. **Given** 사용자가 범위 탭을 전환한 상태, **When** "전체" 탭을 클릭하면, **Then** DB의 전체 회차 데이터 기반 빈도가 표시된다.

---

### User Story 2 - HOT/COLD 번호 실시간 표시 (Priority: P1)

범위 필터에 따라 가장 자주 나온 번호(HOT)와 가장 적게 나온 번호(COLD)가 실제 데이터를 기반으로 표시된다.

**Why this priority**: HOT/COLD 표시는 빈도 데이터와 직접 연결되어 있으며, 사용자가 번호를 선택할 때 핵심 참고 지표이다. User Story 1과 함께 구현되어야 의미가 있다.

**Independent Test**: 선택한 범위의 빈도 데이터를 수동 계산하여 HOT 6개, COLD 6개 번호가 화면에 표시된 번호와 일치하는지 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 통계 페이지에서 특정 범위 탭이 선택된 상태, **When** 빈도 데이터가 로드되면, **Then** 출현 횟수 상위 6개 번호가 HOT 영역에, 하위 6개 번호가 COLD 영역에 표시된다.
2. **Given** "최근 50회" 탭이 선택된 상태, **When** "최근 100회"로 전환하면, **Then** HOT/COLD 번호가 해당 범위 기준으로 갱신된다.

---

### User Story 3 - 역대 회차 데이터 조회 (Priority: P2)

통계 페이지 하단에서 가장 최근 회차부터 역순으로 실제 추첨 결과(당첨 번호, 보너스 번호, 1등 당첨금)를 확인할 수 있다.

**Why this priority**: 역대 회차 목록은 독립적인 정보 제공 기능으로, 빈도 분석 없이도 단독으로 가치가 있다. 다만 HOT/COLD보다는 부가 정보이므로 P2로 설정.

**Independent Test**: 역대 회차 목록에 표시된 회차 번호, 날짜, 당첨 번호가 DB의 실제 데이터와 일치하는지 비교하여 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 사용자가 통계 페이지에 진입한 상태, **When** 페이지가 로드되면, **Then** 가장 최근 8개 회차의 추첨 결과(회차번호, 날짜, 당첨번호 6개, 보너스번호, 1등 당첨금)가 역순으로 표시된다.
2. **Given** DB에 새로운 회차 데이터가 추가된 상태, **When** 사용자가 통계 페이지를 다시 방문하면, **Then** 최신 회차 데이터가 포함된 목록이 표시된다.

---

### Edge Cases

- 네트워크 오류 또는 Supabase 연결 실패 시 어떻게 표시하는가? 화면 중앙에 에러 메시지 텍스트를 표시한다. 별도 재시도 버튼은 제공하지 않으며, 페이지 재진입으로 재조회한다.
- DB에 50회 미만의 데이터만 존재할 때 "최근 50회" 탭은 어떻게 동작하는가? 존재하는 데이터만으로 빈도를 계산한다.
- 모든 번호의 출현 횟수가 동일할 때 HOT/COLD 구분은 어떻게 하는가? 번호 오름차순으로 상위/하위를 결정한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 선택된 범위(최근 50회, 최근 100회, 전체)에 따라 DB(Supabase RPC)에서 빈도를 집계하여 1~45번 각 번호의 출현 횟수 결과만 반환받아야 한다.
- **FR-002**: 시스템은 조회된 빈도 데이터에서 출현 횟수 상위 6개 번호를 HOT으로, 하위 6개 번호를 COLD로 분류하여 표시해야 한다.
- **FR-003**: 시스템은 1~45번 전체 번호의 출현 빈도를 바 차트 형태로 표시해야 한다.
- **FR-004**: 시스템은 최근 8개 회차의 추첨 결과(회차번호, 추첨일, 당첨번호 6개, 보너스번호, 1등 당첨금)를 역순으로 표시해야 한다.
- **FR-005**: 시스템은 데이터 로딩 중 로딩 상태를 사용자에게 표시해야 한다.
- **FR-006**: 시스템은 데이터 조회 실패 시 에러 상태를 사용자에게 표시해야 한다.
- **FR-007**: 범위 탭 전환 시 빈도 데이터와 HOT/COLD 번호가 해당 범위 기준으로 갱신되어야 한다.
- **FR-008**: 시스템은 페이지 진입 시 3개 범위(50/100/전체)의 빈도 데이터와 역대 회차 데이터를 모두 조회하고, 탭 전환 시에는 캐싱된 데이터를 즉시 표시해야 한다. 페이지 재진입 시에는 새로 조회한다.

### Key Entities

- **추첨 회차(Lottery Draw)**: 회차 번호, 추첨일, 당첨번호 6개(오름차순), 보너스번호, 1등 당첨금, 총 판매액
- **번호 빈도(Number Frequency)**: 번호(1~45), 출현 횟수, 조회 범위(50/100/전체)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 통계 페이지의 모든 수치(빈도, HOT/COLD, 역대 회차)가 실제 추첨 데이터와 100% 일치한다.
- **SC-002**: 범위 탭 전환 시 데이터가 2초 이내에 갱신된다.
- **SC-003**: 페이지 초기 로드 시 데이터가 3초 이내에 표시된다.
- **SC-004**: 데이터 로딩 중 사용자가 로딩 상태를 인지할 수 있다.

## Assumptions

- Supabase에 `lottery_draws` 테이블이 이미 존재하고, 충분한 회차 데이터가 적재되어 있다.
- 빈도 계산은 Supabase RPC 함수에서 수행하며, 프론트엔드는 집계된 결과만 수신한다.
- 기존 UI 디자인(레이아웃, 색상, 컴포넌트)은 유지하며 데이터 소스만 교체한다.
- 역대 회차 목록은 최근 8개로 제한한다(현재 하드코딩된 수량과 동일).
- 사용자는 안정적인 네트워크 환경에서 앱을 사용한다.
